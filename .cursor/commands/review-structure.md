# 프로젝트 구조 점검

이 프로젝트는 **HTML · CSS · JavaScript만 사용하는 입문자용 테트리스**입니다.
코드를 수정하지 말고, 현재 저장소 구조와 역할 분리를 점검한 뒤 보고서를 작성하세요.

## 점검 범위

다음 파일을 읽고 분석합니다.

- `index.html` — 마크업·UI 구조
- `style.css` — 레이아웃·보드·블록 스타일
- `script.js` — 게임 상태·렌더링·이벤트
- `README.md` — 문서와 실제 코드 일치 여부

## 점검 항목

### 1. 파일 역할 분리

- HTML은 구조만, CSS는 표현만, JS는 동작만 담당하는지
- 인라인 스타일·인라인 스크립트가 없는지
- DOM id/class 이름이 HTML·CSS·JS에서 일관되는지

### 2. 프로젝트 제약 준수

- 빌드 도구·번들러·외부 라이브러리를 사용하지 않는지
- JavaScript가 ES Modules(`type="module"`)로 연결되는지
- 입문자가 한 폴더에서 바로 실행할 수 있는 단순 구조인지

### 3. 게임 구조 (script.js)

- 상수(`COLS`, `ROWS`, `SHAPES` 등)가 상단에 모여 있는지
- 상태(`state`)와 DOM 참조가 구분되어 있는지
- 아래 역할별로 함수가 나뉘어 있는지
  - **보드 생성**: `createBoard()`, `buildBoardGrid()`
  - **피스 생성**: `createPiece()`
  - **렌더링**: `renderBoard()`, `drawPiece()`, `render()`
  - **이벤트**: `bindEvents()`, `handleStart()`, `resetGame()`
  - **초기화**: `init()`
- 함수마다 JSDoc 주석이 있는지
- 주석·커밋 메시지 규칙(한국어)을 따르는지

### 4. UI 구조 (index.html + style.css)

- 게임 보드, 점수, 시작/재시작 버튼, 조작법 안내가 있는지
- 보드가 **10열 × 20행** CSS Grid로 표현되는지
- I, O, T, S, Z, J, L 블록 색상 클래스가 정의되어 있는지
- 시맨틱 태그(`header`, `main`, `section`, `aside`) 사용이 적절한지

### 5. 문서 일치

- `README.md`의 실행 방법·화면 구성·폴더 구조가 실제 코드와 맞는지
- 구현되지 않은 기능(자동 낙하, 키보드 조작 등)이 문서에 과장되어 있지 않은지
